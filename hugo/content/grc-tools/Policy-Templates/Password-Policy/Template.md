10|## Purpose
11|
12|This policy defines the requirements for creating, managing, and protecting authentication credentials used to access ____'s information systems and applications. It incorporates current best practices from NIST SP 800-63B and prioritizes usability alongside security to promote sustainable password hygiene across the organization.
13|
14|## Scope
15|
16|This policy applies to all Personnel - employees, contractors, temporary workers, and any other individuals who hold accounts on any system that resides within ____'s facilities or has access to the organizational network. It covers all authentication credentials, including passwords, passphrases, PINs, and multi-factor authentication tokens.
17|
18|## Policy
19|
20|### Password Construction Requirements
21|
22|| Requirement | Specification |
23||-------------|---------------|
24|| Minimum length | **12 characters** (NIST SP 800-63B recommends minimum 8 for user-chosen, 12 for stronger assurance) |
25|| Maximum length | At least 64 characters; systems must not truncate passwords |
26|| Character composition | No mandatory complexity rules (no required uppercase, lowercase, digits, or special characters) |
27|| Passphrases | Encouraged; multiple words with spaces are stronger and more memorable than short complex passwords |
28|| Password generation | All passwords should be generated by the organization-approved password manager using maximum available length and character sets |
29|| Banned passwords | Passwords appearing in known breach corpora, dictionary words, keyboard patterns, and context-specific words are prohibited |
30|| Password history | Do not reuse the last ____ (e.g., 5) passwords |
31|| Cross-system reuse | Each system and service must use a unique password. Password reuse across distinct systems is prohibited |
32|
33|### Password Manager Requirement
34|
35|All Personnel must use the organization-approved password manager (____) for generating and storing all work-related credentials. The password manager provides:
36|
37|- Automatic generation of strong, unique passwords for every service
38|- Encrypted storage synchronized across authorized devices
39|- Secure sharing of credentials when necessary (without revealing the password text)
40|- Breach monitoring to alert users when a stored password appears in known data breaches
41|
42|The master password for the password manager is the **single password that Personnel must memorize**. This master password must:
43|- Be at least ____ (e.g., 16) characters
44|- Be a memorable passphrase, not a complex string
45|- Never be written down or stored electronically outside the password manager
46|- Be changed immediately if suspected of compromise
47|
48|### Single Sign-On (SSO)
49|
50|Where technically feasible, systems and applications must be integrated with the organization's Single Sign-On (SSO) provider. SSO reduces the number of credentials Personnel must manage, centralizes access control, and enables consistent enforcement of authentication policies including MFA. New systems acquired or developed must support SAML 2.0, OIDC, or equivalent federation protocols.
51|
52|### Multi-Factor Authentication (MFA)
53|
54|MFA must be enabled for all systems and applications that support it. This includes, at minimum:
55|
56|- All systems accessible from the public internet
57|- All systems containing Restricted or Confidential data
58|- All administrative and privileged access interfaces
59|- SSO, identity provider, and password manager access
60|- Email and communication platforms
61|- Code repositories and CI/CD systems
62|- Cloud infrastructure management consoles
63|
64|**Approved MFA methods** (in order of preference):
65|1. FIDO2 / WebAuthn hardware security keys (passkeys)
66|2. Time-based one-time passwords (TOTP) via authenticator applications
67|3. Push notifications to a trusted device
68|4. SMS or voice call (acceptable only when no stronger method is available; deprecated for new implementations)
69|
70|**Prohibited MFA methods:**
71|- Email-based one-time codes
72|- Security questions as a primary factor
73|- SMS as the sole MFA method for privileged accounts
74|
75|### Credential Distribution and Initial Setup
76|
77|Initial credentials must be unique, non-guessable, delivered through a secure channel, and changed upon first use. The recipient's identity must be verified before issuance.
78|
80|
81|### Password Storage in Systems
82|
83|Applications and systems that store user passwords must:
84|- Use an approved password hashing algorithm: bcrypt, scrypt, Argon2id, or PBKDF2 with HMAC-SHA256.
85|- Apply a unique, cryptographically random salt per password of at least 32 bits.
86|- Use a work factor calibrated to require at least 100ms of computation on current hardware.
87|- Never store passwords in plaintext, reversible encryption, or unsalted hashes.
88|
89|### Credential Sharing
90|
91|Passwords are confidential and must not be shared. If another individual requires access to a system:
92|- Provision a separate account with appropriate access rights.
93|- If shared access is unavoidable, use the password manager's secure sharing feature.
94|- Never share passwords via email, chat, text message, phone, or written notes.
95|
96|### Password Rotation
97|
98|**Passwords must NOT be rotated on a fixed schedule** unless there is evidence or suspicion of compromise. This aligns with NIST SP 800-63B guidance.
99|
100|Passwords must be changed immediately when:
101|- There is evidence or reasonable suspicion of compromise.
102|- A breach of the system or service has been publicly disclosed.
103|- The password is discovered to have been stored insecurely.
104|- The password is found in a known breach corpus.
105|
106|### Password Compromise Response
107|
108|If a password is suspected of compromise, it must be changed immediately on the affected system and any other system where it was reused. The incident must be reported to the Security Officer and account activity must be reviewed for signs of unauthorized access.
109|
111|
112|### Service Account and Machine Credentials
113|
114|Service accounts, API keys, and machine-to-machine credentials:
115|- Must use randomly generated credentials of at least ____ (e.g., 32) characters or their equivalent entropy.
116|- Must be stored in an approved secrets management system.
117|- Must never be embedded in source code, configuration files, or deployment scripts.
118|- Must be rotated regularly as defined in the Encryption Policy key rotation schedule.

120|## Roles and Responsibilities
121|
122|| Role | Responsibility |
123||------|----------------|
124|| ____ (e.g., CISO / Security Officer) | Policy owner; annual review; approve authentication exceptions |
125|| IT / Identity and Access Management | Provision accounts; manage SSO integrations; enforce MFA enrollment |
126|| Engineering | Implement approved password hashing in applications; integrate with SSO and secrets management |
127|| All Personnel | Use the approved password manager; generate unique passwords; enable MFA everywhere; report compromised credentials immediately |
128|
129|## Compliance and Enforcement
130|
131|Compliance is verified through automated MFA coverage checks, password manager adoption audits, periodic review of authentication configurations, and breach corpus checks. Personnel found to have violated this policy - particularly through credential sharing, reuse, or insecure storage - may be subject to disciplinary action.
